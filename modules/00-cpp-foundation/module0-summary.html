📦 知识点详解
0.1
Build.cs / Target.cs
入门
Target.cs 定义编译目标（Game/Editor），Build.cs 定义模块依赖（Public/Private）。
  • TargetType：Game（打包）/ Editor（开发）/ Client / Server
  • Public 依赖：头文件中 #include 了 → 传递性继承
  • Private 依赖：只在 .cpp 中用 → 不暴露给使用者
  • PCHUsage：UseExplicitOrSharedPCHs 最常用
  • PublicDefinitions：向依赖方传递宏定义
  • 条件依赖：if (Target.bBuildEditor) 编辑器专属模块
// Build.cs 核心 PublicDependencyModuleNames.AddRange(new string[] { "Core", "CoreUObject", "Engine", "GameplayTags" }); // Target.cs 核心 Type = TargetType.Editor; ExtraModuleNames.Add("StudyProject");
⚠ 踩坑改了 Build.cs 后增量编译可能不生效，必须清 Intermediate\Build 全量重编
0.2
插件结构 .uplugin
入门
插件 = .uplugin 描述文件 + 一个或多个模块。模块通过 IModuleInterface 管理生命周期。
  • .uplugin：JSON 格式，声明模块列表、类型、加载时机
  • IModuleInterface：StartupModule / ShutdownModule 生命周期
  • IMPLEMENT_MODULE：注册宏，没有它模块无法加载
  • Public/ 目录：对外暴露的头文件 + XXX_API 导出宏
  • Private/ 目录：内部实现，外部不可见
  • 主项目依赖插件：Build.cs 加插件模块名 + .uproject 启用
// .uplugin 模块声明 "Modules": [{ "Name": "StudyPlugin", "Type": "Runtime" }] // 模块注册宏 IMPLEMENT_MODULE(FStudyPluginModule, StudyPlugin)
0.3
Module 划分 Runtime/Editor
进阶
把插件拆成 Runtime 模块（打包也编译）和 Editor 模块（仅编辑器编译），依赖方向单向。
模块类型编译时机能依赖
Runtime编辑器 + 打包只能依赖 Runtime
Editor仅编辑器Runtime + Editor
DeveloperDev/StagingRuntime + Developer
⚠ 铁律Editor → Runtime ✅   Runtime → Editor ❌（打包编译失败）
  • PrimaryGameModule：.uproject 中 Modules[0]，游戏入口
  • ToolMenus：注册编辑器工具栏按钮
  • FAppStyle：UE5.7 替代废弃的 FEditorStyle
0.4
第三方库集成
进阶
把 C/C++ 静态库集成到 UE 模块中，三步走：头文件路径 → 链接 .lib → Wrapper 包装。
// Build.cs 第三方库配置 string LibDir = Path.Combine(ModuleDirectory, "..", "ThirdParty", "SimpleMathLib"); // 1. 头文件路径 PublicIncludePaths.Add(Path.Combine(LibDir, "include")); // 2. 链接 .lib PublicAdditionalLibraries.Add(Path.Combine(LibDir, "lib", "SimpleMathLib.lib"));
  • extern "C"：防止 C++ name mangling 导致链接失败
  • Wrapper 模式：封装 C API 为 UE 风格的类
  • 静态库：代码直接编入模块，打包自动包含
  • 动态库：额外需要 RuntimeDependencies 拷贝 .dll
  • 多平台：用 Target.Platform 选择对应平台的库
0.5
Live Coding / Hot Reload
入门
Live Coding 秒级热重载函数体（不丢状态），Hot Reload 完整重编模块（可加反射属性但丢状态）。
Live CodingHot Reload
速度秒级较慢
改函数体✅✅
加 UPROPERTY❌✅
运行时状态保留重置
改 Build.cs❌❌
改函数体 → Live Coding (秒级) → 不够？ → 加反射属性 → Hot Reload → 不够？ → 改 Build.cs → 清缓存全量重编

🚨 踩坑总结
!
#if vs #ifdef
UE5 开启 /we4668，#if MACRO 在宏未定义时会报 C4668 错误。
解决用 #ifdef MACRO 替代 #if MACRO
!
Build.cs 增量缓存
修改 Build.cs 加新依赖后，UBT 增量编译不重新解析依赖，导致 LNK2019。
解决删 Intermediate\Build → 重新生成项目文件 → 全量编译
!
FEditorStyle 废弃
UE5.7 中 FEditorStyle::GetStyleSetName() 已废弃，编译报 C4996。
解决改用 FAppStyle::GetAppStyleSetName()，#include "Styling/AppStyle.h"
⚡ UBT 编译命令速查
# 变量 $UBT = "H:\UE_Engine\UE_5.7\Engine\Binaries\DotNET\UnrealBuildTool\UnrealBuildTool.exe" $PROJ = "H:\UE_Project\Study\StudyProject\StudyProject.uproject" # 1. 常规增量编译（改了 .h/.cpp 但没改 Build.cs） & $UBT StudyProjectEditor Win64 Development -project=$PROJ -waitmutex # 2. 改了 Build.cs 后（加/删依赖、改 PCH、加宏定义） Remove-Item "...\Intermediate\Build" -Recurse -Force & $UBT -projectfiles -project=$PROJ -game -engine & $UBT StudyProjectEditor Win64 Development -project=$PROJ -waitmutex # 3. 重新生成项目文件 & $UBT -projectfiles -project=$PROJ -game -engine
🔄 模块生命周期详解 (IModuleInterface)
源码：Engine/Source/Runtime/Core/Public/Modules/ModuleInterface.h。模块 DLL 加载后，引擎按以下顺序调用 5 个回调：
// 1. DLL 加载后、模块对象创建后立即调用 // 适合：加载依赖模块、注册委托、初始化数据 virtual void StartupModule() {} // 2. 所有模块 Startup 后调用 // 适合：需要等依赖模块初始化完成后的操作 virtual void PostLoadCallback() {} // 3. 卸载前调用（还在正常运行状态） // 适合：提前保存状态、通知其他模块 virtual void PreUnloadCallback() {} // 4. 模块即将卸载、对象即将销毁前调用 // 适合：注销注册的菜单、委托等 // ⚠️ 按 StartupModule 的逆序调用 —— 保证依赖模块还没被卸载 virtual void ShutdownModule() {} // 控制方法 virtual bool SupportsDynamicReloading() { return true; } // 是否支持热重载 virtual bool SupportsAutomaticShutdown() { return true; } // 退出时自动关闭 virtual bool IsGameModule() const { return false; } // 是否游戏模块
加载 DLL → new FModule() → StartupModule → PostLoadCallback → 运行中... → PreUnloadCallback → ShutdownModule → delete + 卸载 DLL
默认实现类IsGameModule()用途
FDefaultModuleImplfalse引擎/插件模块，不需自定义生命周期
FDefaultGameModuleImpltrue游戏模块，不需自定义生命周期
🏷️ IMPLEMENT_MODULE 详解
位置
放在哪里？
放在模块的某个 .cpp 文件末尾、全局作用域中。一个模块只能有一个 IMPLEMENT_MODULE。
// StudyPluginModule.cpp 末尾 // ... 代码实现 ... // ━━━ 末尾，全局作用域 ━━━ IMPLEMENT_MODULE(FStudyPluginModule, StudyPlugin)
原理
宏展开做了什么
生成模块工厂函数 + 注册 FModuleInitializerEntry 静态对象。
// 宏展开后（简化版） static IModuleInterface* InitializeStudyPluginModule() { return new FStudyPluginModule(); // 创建实例 } static FModuleInitializerEntry StudyPluginInitializerEntry( TEXT("StudyPlugin"), // 模块名 InitializeStudyPluginModule // 工厂函数 );
宏用途IsGameModule()
IMPLEMENT_MODULE非游戏模块（引擎、工具、插件）false
IMPLEMENT_GAME_MODULE游戏模块true
关键第二个参数（模块名）必须与 .uplugin / .uproject / Build.cs 中的名字完全一致，否则 monolithic build 下会断言失败。
🏗️ 模块管理与 PrimaryGameModule
管理
模块管理的 6 个维度
维度在哪里
声明模块存在.uproject / .uplugin → Modules 数组
编译依赖Build.cs → Public/PrivateDependency
编译目标Target.cs → ExtraModuleNames
模块入口.cpp 末尾 → IMPLEMENT_MODULE
生命周期继承 IModuleInterface

入口
PrimaryGameModule
.uproject 中 Modules[0] 就是 PrimaryGameModule（主游戏模块）。
// .uproject { "Modules": [ { "Name": "StudyProject" } // ← Primary ] }
  • 自动被 Target.cs 引用（不用手动写 ExtraModuleNames）
  • 是游戏入口，UHT 优先扫描
  • 其他游戏模块在 Build.cs 中依赖它
  • 名字必须与 Target.cs 中的 ExtraModuleNames 一致
⚙️ 模块类型 (Type) 与加载时机 (LoadingPhase)
Type编译时机能依赖典型用途
Runtime编辑器 + 打包RuntimeActor、Component、游戏逻辑
Editor仅编辑器Runtime + Editor编辑器工具、Detail 自定义
DeveloperDev/StagingRuntime + Developer开发工具、性能分析器
ServerOnly仅服务端Runtime + Server专用服务器逻辑
ClientOnly仅客户端Runtime + Client客户端专属逻辑
LoadingPhase时机典型用途
PostConfigInit最早，配置加载后需要读配置的底层模块
PreDefault大多数模块之前需要被其他模块依赖的底层模块
Default正常时机（最常用）90% 的模块
PostDefault大多数模块之后需要等所有 Default 加载完
PostEngineInit引擎完全初始化后不影响启动速度的模块
⚠ 依赖铁律Editor → Runtime ✅  |  Runtime → Editor ❌（打包时 Editor 模块不存在，链接失败）

🎨 FAppStyle — UE5 统一样式系统
